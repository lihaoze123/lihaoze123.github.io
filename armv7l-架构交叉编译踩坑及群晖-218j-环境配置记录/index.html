armv7l 架构交叉编译踩坑及群晖 218j 环境配置记录

记录配置群晖 218j (armv7l 架构) 常用环境，仅作为记录。关于交叉编译网络上的资料比较少，很多地方踩了很多坑，有一些心得，同时群晖自带的套件中心套件很少，本人的型号没有什么第三方库，想要装一个 aria2 网络上能查到的资料都很少，且 DSM7.0 之后本来能用的方法也失效了，即使装了 Entware ，很多包版本也是落后的，只能自己编译，所以本文可能有参考价值。但 这不是一篇参考指南，编译等操作本就门槛不低，而且相比常规的编译，armv7l 架构不管是本地编译还是交叉编译都非常抽象，比较基本的操作我也都省略了，也懒得截图。很多地方不一定是标准操作，仅供参考。

本人交叉编译时踩的坑

  • 有调试编译了一整天，最后发 issue 才知道不支持交叉编译的（不写在文档里，issue 里也没有相关的说话）。

  • 也有搞了半天依赖都编译不出来的，结果人家开发者最近已经不用这个依赖，发过 pull request 了的。

  • 群晖自带的内核版本太低, 有的即使调试一整天，系统调用也可能会报错。

总之交叉编译，特别是给 armv7l 编译是一件很麻烦的事，网上几乎查不到资料，下文给出的配置文件 (针对 armv7l) 对于大部分软件应该是适用的，但是具体到每个软件里还得看情况，即使都用的是 make，configure，cmake，但是具体到参数里也不一定通用，如果条件允许尽量本地编译。

对于莫名其妙的问题，善用 StackOverFlow 和 Google 或许会有帮助，或者可能是项目本身的问题，尽量下载最近的 release 进行编译，很多时候直接编译 git master 不一定是好选择 。
